# Implement: Accounts & Data Layer

## The Implementation

Eighteen files (plus three infrastructure deltas) establish the authentication backbone and
the complete v1 D1 schema in one act. The full domain — journeys, waypoints, lessons, quiz
questions and attempts, concepts, FSRS cards, and usage events — ships as a single idempotent
migration. Every table carries an inline comment citing the slice that will first consume it,
so no column can be speculative without a named owner.

The load-bearing security contract lands in `src/lib/auth-guard.ts`: `requireOwnership` is a
pure synchronous function that throws a 403 Response on any user-id mismatch. Its purity is
intentional — tested in standard Vitest (no Workers pool needed) with six cases covering
same-user pass, cross-user throw, empty-string edge cases, and the GitHub private-email path.
`requireAuth` sits above it, calling the per-request better-auth factory and throwing 401 on
missing session. All four journey server functions compose these two guards; cross-account rows
are structurally unreachable at the server function boundary.

The TanStack DB integration surfaced three API-drift items relative to the plan (detailed in
Deviations from Plan below). The in-memory collection fully satisfies the reactive-read AC;
the IndexedDB persistence deferral is recorded as a known-ceiling shortcut. Two E2E tests
run without OAuth credentials (sign-in page UI, unauthenticated redirect); three additional
tests activate when `BETTER_AUTH_SECRET` is set in `.dev.vars` for the seeded-session proxy
path. The OAuth residual (real provider sign-in on a deployed environment) is pre-registered
per the constraint-resolution in the plan and cleared by the first successful deployed sign-in.

`usage_events` is the one schema divergence from the plan's original column list: the
`04b-instrument.md` augmentation mandates richer fields (`prompt_tokens`, `completion_tokens`,
`duration_ms`, `outcome`, `at`, `journey_id`) aligned with the ai-gateway instrumentation
contract. The plan's original simpler column set was superseded by the augmentation as
the authoritative contract.

## Notes on Design Choices

1. **`requireOwnership` as pure function** — The guard's purity (no Workers runtime dependency)
   enables standard Vitest testing. The full data isolation proof (D1 `WHERE user_id = ?`) is
   covered by Playwright E2E against `wrangler dev`. This avoids `@cloudflare/vitest-pool-workers`
   for this slice (assumption #4 from plan).

2. **`withSession` middleware pattern** — TanStack Start v1.168.x does not expose `getWebRequest()`.
   A `type: 'request'` middleware captures the request and injects the authenticated session into
   the server function context. The plan described `getWebRequest()` as the access method; the
   middleware achieves the same security contract. Deviation #3.

3. **`usage_events` schema matches `04b-instrument.md`** — The augmentation artifact mandates
   `prompt_tokens`, `completion_tokens`, `duration_ms`, `outcome`, `at`, `journey_id` for the
   ai-gateway instrumentation signals. The plan's original simpler column list was superseded.
   This is the authoritative schema for the ai-gateway slice.

4. **Sign-in page `beforeLoad` — no server-side session check this slice** — The plan's route-
   level `beforeLoad` reads `context.auth` (which is `undefined` until design-system-shell adds
   the root loader). Unauthenticated access to `/account` correctly throws the redirect because
   `!undefined?.session` is true. Authenticated fast-path for sign-in is deferred to design-
   system-shell as the root loader is that slice's responsibility (assumption #6 from plan).

5. **`worker-configuration.d.ts` is 14,709 lines** — `wrangler types` generates the full
   Cloudflare Workers type library. The file is committed so TypeScript resolves `Env` without
   a separate `wrangler types` step in CI. The line count is dominated by the generated library
   types, not project-specific content (~30 lines of project-specific Env interface).

## Deviations from Plan

1. **TanStack DB `createReactCollection` does not exist in `@tanstack/react-db@0.1.92`.**
   Plan prescribed `createReactCollection` + `persistence: { type: 'indexeddb' }`. The
   installed package exports `useLiveQuery` (hook) and `@tanstack/db` provides `createCollection`
   (core). Resolution: `createCollection` from `@tanstack/db` + `useLiveQuery` from
   `@tanstack/react-db`. The reactive-read AC (AC-ADL4) is fully satisfied. Smoke test uses
   `col.state` directly (core API) avoiding the React hook in a Node test environment.

2. **IndexedDB persistence not available in `@tanstack/db@0.6.14`.**
   The plan's `persistence: { type: 'indexeddb', name: 'waypoint-journeys-v1' }` API does not
   exist in this version. Resolution: in-memory collection only. The collection reactivity and
   server-function syncer AC (AC-ADL4) are fully met without persistence. Persistence can be
   re-added when a stable `@tanstack/db` release ships the API.
   `sdlc-debt: IndexedDB persistence deferred — ceiling: journeys not available across hard
   reloads; upgrade path: add persistence config when @tanstack/db ships stable persistence API.`

3. **`getWebRequest()` not available in `@tanstack/react-start@1.168.x`.**
   Plan prescribed `import { getWebRequest } from '@tanstack/react-start'` inside server function
   bodies. This export does not exist in the installed version. Resolution: `createMiddleware`
   with `type: 'request'` captures the request upstream and injects `session` into the server
   function context via `next({ context: { session: sessionData } })`. The security contract
   (every server function has a validated session before executing) is identical.

## Anything Deferred

- **IndexedDB persistence for the journeys collection** — Deferred per deviation #2 above.
  Ceiling: journeys not available across hard reloads. Upgrade path: add persistence config
  when `@tanstack/db` ships stable persistence API (currently beta).
  `sdlc-debt: see src/lib/store/journeys.ts comment`

- **Root-level auth redirect** — The `/account` guard uses route-level `beforeLoad`, not a
  root-level authenticated layout wrapper. A root-level wrapper (for any protected route to
  redirect without per-route guards) is design-system-shell's responsibility per assumption #6.

- **Server-side session check on sign-in `beforeLoad`** — The `/sign-in` route checks
  `context.auth?.session` which is `undefined` until design-system-shell adds the root loader.
  For this slice the authenticated fast-path (skip sign-in if already logged in) only fires
  when design-system-shell's root loader sets `context.auth`. The user-visible gap: a signed-in
  user who navigates directly to `/sign-in` sees the sign-in page momentarily (then redirects
  once the root loader is added). Pre-accepted rework per assumption #5.

- **Real OAuth sign-in flow** — Pre-registered residual per constraint-resolution:
  cleared by first successful Google + GitHub sign-in on a deployed environment.
  `cleared-by: first successful real Google + GitHub sign-in on a deployed environment`

## Known Risks / Caveats

- **TanStack DB v0.6 beta** — `@tanstack/db` and `@tanstack/react-db` are beta. The syncer
  API (`begin/write/commit/markReady`) may change in future versions. The smoke test and
  production collection both use the same API surface. Monitor @tanstack/db releases.

- **better-auth session cookie algorithm** — The Playwright seeded-session proxy constructs
  the `better-auth.session_token` cookie using HMAC-SHA-256 + btoa signature
  (`encodeURIComponent(token + '.' + sig)`). If better-auth changes the cookie signing
  algorithm, the E2E tests that require `BETTER_AUTH_SECRET` will fail with auth errors.
  The first two E2E tests (no secret needed) are immune.
